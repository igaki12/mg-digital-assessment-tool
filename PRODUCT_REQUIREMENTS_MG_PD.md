# 次世代デジタルバイオマーカーアプリ開発要件定義書

ご提示いただいたメールのやり取り結果および関連文献、特に「BioDigit MG」や「ME&MGopen」などの先行するデジタルバイオマーカーアプリの事例や音声分析手法に関する知見を統合し、作成するデジタルバイオマーカーアプリの要件を以下のとおり整理しました。

## Ⅰ．アプリの基本コンセプト・位置づけ

### 目的・役割

神経学的所見（表情、姿勢、眼瞼下垂、構音障害など）を客観的かつ定量的に評価するデジタルバイオマーカーとして位置づけます。臨床医の評価を代替するものではなく、あくまで補助する役割を担うものとします [1]。

### 評価の主眼

患者間の比較よりも、同一患者における経時的変化、すなわち日内変動や治療介入前後の変化を高い信頼性で捉えることを最重要目的とします [1], [2]。

### 利用環境

特殊なウェアラブル機器は必要とせず、患者自身のスマートフォンに搭載されたカメラとマイクを用いて、外来・病棟のみならず自宅でも単独で計測可能な構成とします [1], [2], [3]。

## Ⅱ．対象疾患と検査項目・評価対象

### 想定疾患

対象疾患は、重症筋無力症（MG）、パーキンソン病、およびパーキンソン症候群などとします [1]。

### 評価対象（検査項目）

既存の画像・動画ベースの評価項目に加え、新たにマイクを用いた「音声（構音障害・発声障害）」の評価項目を追加します。

- 姿勢
  - 体幹前屈角（camptocormia 相当）
  - 首下がり角
  - 側方偏位
  - 正面・側面から取得した画像または動画を用いて代表値を算出します [1], [2]。
- 表情
  - 仮面様顔貌
  - 瞬目 など [1]
- 眼症状
  - 眼瞼下垂 [1], [4]
- 音声（追加要件）
  - MG やパーキンソン病に伴う構音障害（dysarthria）や発声異常を評価対象とします [4], [5]。

## Ⅲ．音声機能（追加要件）の具体的な計測・分析イメージ

先行する MG 向けアプリ（BioDigit MG 等）や音声分析（Praat 等）の知見を参考に、以下の機能を実装します。

### 発話タスクの録音

- 数字のカウント（例: 1 から 50 まで）
  - 構音の正確さ、流暢性、発話速度の低下、すなわち疲労の影響を評価します [4], [6]。
  - 画面上の点滅に合わせて発声させるなど、発話を支援するガイド機能を設けます [6]。
- 定型文の音読
  - 明瞭さや自然な音声パターンを評価します [6]。
- 持続母音（「あー」など）の発声
  - 発声の安定性を評価します [7]。

### 音声の定量分析（バックエンド処理）

取得した音声データから、以下のような定量指標を算出します [8], [9], [10]。

- 基本周波数（F0 / ピッチ）
- 声の強さ（loudness / intensity）
- 周波数のゆらぎ（Jitter）
- 振幅のゆらぎ（Shimmer）
- 調波対雑音比（HNR）

## Ⅳ．計測方法・実施フローに関する要件

### 測定のタイミング

単回測定だけでなく、MG の日内変動や治療効果を捉えるため、**「朝・夕などの時間帯指定測定」や「症状増悪時のイベント駆動測定」** に対応します [2]。

### UI / UX の最適化

高齢者や神経疾患患者でも負担なく操作できるよう、手順は可能な限り簡素化します [3]。

### 経時比較 UI

患者自身および診療時の医療者が、同一患者内の経時的変化を短時間で把握できるよう、`Records / 記録を見る` 画面に以下の要件を持たせます。

- 検査種別ごとにタブを分け、指標が混在しないようにします。
- 代表値を切り替えながら、`日毎の変動` と `日内変動` を確認できるようにします。
- `日毎の変動` では、平均値だけでなく最大値・最小値を表示し、その間を帯で示します。
- `日毎の変動` の集計粒度は `日 / 週 / 月` を切り替え可能とします。
- `日内変動` では、本日と直近1週間平均を重ねて表示し、患者内でのリズムのずれを確認しやすくします。
- 非モバイル画面では、変動グラフを上段に広く取り、その下に履歴一覧と詳細を配置します。
- モバイル画面では、履歴 row のタップで詳細モーダルを開き、表示中は背景スクロールを停止します。

## Ⅴ．データの種類・解析に関する要件

### 保存データ

算出した数値データ（角度、音声特徴量など）のみならず、介入前後の状態を視覚的・聴覚的に振り返ることができるよう、画像・動画データおよび音声データそのものもあわせて保存します [2]。

- セッションメタデータには、検査種別、日時、代表スコアを保持します。
- 時系列データには、フレーム単位またはタスク単位の数値列に加え、問診票内訳や検査サマリーなどの補足詳細を保持できる構造を持たせます。
- 音声検査では、録音 Blob と各タスクのメトリクスを別管理できるようにします。

### 解析の考え方

衣服や撮影環境によるばらつき、すなわちアーチファクトを考慮し、完全な絶対値精度よりも再現性と時系列での比較可能性を重視したアルゴリズムとします [2]。

### 臨床との対応

デジタル指標が、実際の神経内科医の評価や既存の臨床スコア（QMG スコアの構音障害項目や MG-ADL など）と関連し、その妥当性を検証できる設計とします [11], [3]。

## Ⅵ．システム・サーバー・セキュリティ要件

### 認証と同意

フロントエンド全体にログイン認証と利用規約への同意フローを実装し、研究倫理、すなわちインフォームドコンセントの枠組みにそのまま組み込めるようにします [3]。

### サーバー構成

スタンドアローンでの動作を基本としつつ、画像・動画・音声といった大容量データを扱うため、クラウドサーバー（XServer VPS など）を用いたアップロードおよび管理機能を備える構成とします [3]。

### 導入の現実性

病院の古い PC 環境でも特別なソフトウェアをインストールすることなく、クラウド経由でデータ確認・運用ができるようにします [3]。

### セキュリティ

通信の暗号化とアクセス制御を徹底し、個人情報および医療データの漏洩リスクを最小化します [3]。
